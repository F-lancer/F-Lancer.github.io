模块化

前端范式之一，主流的代码组织方式。

演变过程

文件划分 (script标签) —— 命名空间的方式 —— IIFE —— 引入模块化规范

文件划分，引入全局导致了全局污染，依靠约定。

命名空间，文件划分 的基础上用对象包裹，解决全局污染，没有私有空间，仍可以被外部访问，没有依赖关系。

利用 IIFE 立即执行函数的方式，

  1. 封装私有空间，作用域。
  2. 向外暴露成员挂载到全局对象 window 上。
  3. 利用闭包具有私有变量。
  4. 利用自执行函数的参数作为依赖执行。

模块化 规范

模块化标准 + 模块加载器（实现自动加载）

Common.js

Node提出的一个标准，它约定了以下规范：

  1. 一个文件一个模块。
  2. 每个模块都有一个单独的作用域。
  3. 通过 module.exports 导出成员。
  4. 通过 require 函数载入模块。

Common JS 是以同步模式加载模块，node 的执行机制是启动时加载模块，使用再引入，而浏览器端不能使用，同步模式请求过大会引起性能问题。

AMD

Asynchronous Module Definition 异步模块化规范，同期 Require.js 库就实现了该规范，并实现了模块加载器的功能。

缺点：使用复杂，模块 JS 文件请求频繁

CMD

CMD，通用模块定义规范，sea.js 实现了该规范。它的出现时为了降低学习成本。

最佳实践

Node环境中遵循 CommonJS 规范，浏览器环境遵循 ES Modules 规范。

CommonJS，是内置模块没有兼容等问题。

ES Modules，在新语法中实现，偶尔会有新语法 兼容问题。

ES Modules

准备工具：

基本特性

  1. script 标签设置 type="module"相对于普通的 script 标签会有不同的特性

    • 自动采用严格模式，忽略 use strict。

  2. 每个 module 都是运行在单独的私有作用域当中。

  3. 外部JS文件是通过 CORS 方式请求的，不在同源地址下，响应需要提供有效 CORS 标头，CORS 标头不支持文件请求。

  4. ESM 的 script 会自动的延迟执行脚本，等同于 script 中的 defer 属性。script 会阻塞代码的执行，延迟执行，使得渲染完成之后再执行脚本 。

导出导入

  1. export 关键字导出。
  2. import 关键字导入。

ESM 兼容性问题

ESM in Browser

IE 不支持 ESM：

  1. babel 转义为 ES5。
  2. polyfill 兼容方案
    • npm install browser-es-module-loader 。
    • npm install promise-ployfill --save-exact。
    • 给 script 标签 nomodule 属性，使其仅在不支持module的浏览器执行
ESM in Node.js

node > 8.5 可以直接使用 ESM。

  1. index.js 改为 index.mjs
  2. node --experimental modules index.mjs 运行试验特性

与 CommonJS 模块交互

  1. ESM 中可以导入 CommonJS 模块。
  2. CommonJS 中不能导入 ESM模块。
  3. CommonJS 始终只会导出一个默认成员。
  4. 注意 import 不是解构导出对象。

在 node 中使用 ESM 和 CommonJS 差异

注意：导入导出可以被 import和exports 代替，__filename 和 __dirname 需要别的方式代替

新版本支持情况

node 12.~：

  1. 不需要改文件后缀名，需要将package.json中添加字段 "type": "module"。
  2. 而再想使用 CommonJS 规范，需要将 common.js 改为 common.cjs。

早期 node 兼容方案

利用 babel：

  1. npm install @babel/node @babel/core @babel/preset-env --save-dev。

  2. 运行 babel-node 命令。

  3. preset-env 插件集合。babel-node index.js --preset=@babel/preset-env

  4. 配置.babelrc 文件，"presets": ["@babel/preset-env"]，presets 是一组插件。

  5. 使用单独的插件

  • 配置.babelrc 文件，"plugins": ["@babel/plugin-transform-modules-commonjs"]

模块化打包工具

主要解决问题：

  1. 环境兼容问题。比如编译转化各种各样的代码。
  2. 将模块化零散的文件打包，减少请求。
  3. 支持其他前端资源模块化。

概要

webpack：

  1. 模块打包器打包。
  2. 模块加载器。
  3. 代码拆分。（避免打包文件过大）
  4. 将资源模块化引入。

模块化工具是解决的是前端整体的模块化。

常用加载器分类

编译转换类：这种类型的 loader 会将加载到的资源模块，转换为 JS 代码。比如：CSS-loader。

文件操作类型：将加载到的资源模块，拷贝到输出目录，同时会将访问路径向外导出。比如：file-loader。

代码检查类：将加载到的代码资源进行校验，统一风格，提高代码质量。

核心工作原理

  1. 从打包入口开始，根据代码中出现的 import require等模块语句，解析推断出模块之间的依赖关系。
  2. 形成依赖关系的依赖树。
  3. 遍历，递归这个依赖树，找到对应的资源文件。
  4. 根据配置文件的 rules 属性，找到对应的模块的加载器加载该模块。
  5. 将加载结果放到输出文件中，实现整个项目的打包。

SourceMap

问：运行代码和源码不同，将难于调试，因此引入 Source Map。用来映射运行代码和源代码之间的关系。

映射一般在 min.map 格式文件夹下。

添加注释：//# sourceMappingURL-jquery-3.4.1.min.map 就会自动请求该map文件。根据该内容逆向解析出源文件。

webpack.config.js 配置方式：

devtool 可以配置很多模式，生成sourceMap文件的方式有很多种，每种方式的效率和效果不同。

大致规律如下：

  1. eval - 是否使用 eval 执行模块代码。
  2. cheap - Source Map 是否包含行信息。
  3. module - 是否能够得到 Loader 处理之前的源代码。
  4. inline - sourcemap以 data-url 的方式嵌入到代码当中。
  5. hidden - 生成但是未引入该文件，比如：jQuery等。
  6. nosources - 提供位置，但不提供源代码，保护生产环境中的代码。

公司项目中使用的是 eval-source-map 模式，错误可以定位到行和列。

https://webpack.js.org/configuration/devtool/

输出文件名 Hash

前端部署前端资源时，会启用服务器的静态资源缓存，对于用户的浏览器而言就会缓存住静态资源，从而减少请求。

问题：设置缓存的时间不能过短，过短就会失去了缓存的意义，时间过长服务器更新资源不能得到及时更新。

在生产环境下，给文件名使用 Hash 值，当资源更新时，文件名也会随之变化，对于客户端就会对新的文件进行请求，从而解决缓存的问题，将时间设置的长一点也无所谓了。

设置哈希的三种方式：

  1. filename: '[name]-[hash].bundle.js' 给 filename 设置项目级别的hash，任意代码更新，全部hash都会发生变化。
  2. filename: '[name]-[chunkhash].bundle.js' chunk块级别的打包，根据打包的模块进行块更新。
  3. filename: '[name]-[contenthash].bundle.js' 设置文件级别的hash，文件改变就会改变hash值。

[contenthash:8] 指定hash的长度，控制缓存推荐该 hash 模式。
